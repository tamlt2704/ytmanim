# Chapter 14: The Monorepo That Takes 20 Minutes to Build

*In which you learn that duplicated types are a lie waiting to happen, and monorepos are the cure (with their own side effects).*

## The Incident

Linus is reviewing a bug report. A user submitted a job with `priority: "URGENT"`. The frontend accepted it. The backend rejected it with a 400 error. The user saw a generic "Something went wrong" message.

> **@Linus:** The backend added `URGENT` as a priority level last sprint. The frontend still has the old enum: `LOW | NORMAL | HIGH | CRITICAL`. The Java `JobPriority` enum and the TypeScript `JobPriority` type are defined in two different repos, by two different teams, and they've drifted apart.

He pulls up the two files side by side:

```java
// job-engine/src/main/java/.../JobPriority.java (backend)
public enum JobPriority {
    LOW, NORMAL, HIGH, CRITICAL, URGENT  // ← added last sprint
}
```

```typescript
// job-engine-dashboard/src/types/api.ts (frontend)
export type JobPriority = 'LOW' | 'NORMAL' | 'HIGH' | 'CRITICAL';
// ← URGENT is missing. Nobody updated this.
```

Two sources of truth. One is wrong. This is the type drift problem, and it gets worse every sprint.

## The Problem — Separate Repos, Duplicated Types

```
repo: job-engine/              (Java, Spring Boot)
  └── JobPriority.java         → LOW, NORMAL, HIGH, CRITICAL, URGENT

repo: job-engine-dashboard/    (React, TypeScript)
  └── types/api.ts             → LOW, NORMAL, HIGH, CRITICAL  ← stale
```

Every time the backend changes a type, someone has to manually update the frontend. "Someone" is nobody. The types drift. Bugs happen. TicketMaster files tickets.

## The Fix — Monorepo with Shared Types

### Step 1: Monorepo Structure

```
job-engine-platform/
├── apps/
│   ├── backend/                 ← Spring Boot job engine (Java)
│   │   ├── src/
│   │   ├── build.gradle
│   │   └── ...
│   └── dashboard/               ← React dashboard (TypeScript)
│       ├── src/
│       ├── package.json
│       ├── vite.config.ts
│       └── ...
├── packages/
│   └── shared-types/            ← generated TypeScript types
│       ├── src/
│       │   └── index.ts         ← auto-generated from OpenAPI
│       ├── package.json
│       └── tsconfig.json
├── turbo.json
├── package.json
└── pnpm-workspace.yaml
```

### Step 2: OpenAPI Spec → TypeScript Types

The job engine already has an OpenAPI spec (Spring Boot generates it from annotations). Use it to generate TypeScript types automatically.

```yaml
# apps/backend/src/main/resources/openapi.yaml (generated by springdoc)
components:
  schemas:
    JobPriority:
      type: string
      enum: [LOW, NORMAL, HIGH, CRITICAL, URGENT]

    Job:
      type: object
      properties:
        id:
          type: string
        name:
          type: string
        priority:
          $ref: '#/components/schemas/JobPriority'
        status:
          $ref: '#/components/schemas/JobStatus'
        submittedAt:
          type: string
          format: date-time
        # ... all fields
```

Generate TypeScript types from the spec:

```bash
npm install -D openapi-typescript
```

```json
// packages/shared-types/package.json
{
  "name": "@job-engine/shared-types",
  "version": "1.0.0",
  "main": "src/index.ts",
  "scripts": {
    "generate": "openapi-typescript ../backend/src/main/resources/openapi.yaml -o src/generated.ts",
    "build": "tsc"
  }
}
```

```typescript
// packages/shared-types/src/generated.ts (auto-generated — don't edit)
export interface components {
  schemas: {
    JobPriority: 'LOW' | 'NORMAL' | 'HIGH' | 'CRITICAL' | 'URGENT';
    JobStatus: 'PENDING' | 'RUNNING' | 'COMPLETED' | 'FAILED' | 'CANCELLED' | 'TIMED_OUT';
    Job: {
      id: string;
      name: string;
      priority: components['schemas']['JobPriority'];
      status: components['schemas']['JobStatus'];
      submittedAt: string;
      startedAt: string | null;
      completedAt: string | null;
      failureReason: string | null;
      dependsOn: string[];
      tags: string[];
    };
    JobMetrics: {
      submitted: number;
      completed: number;
      failed: number;
      activeJobs: number;
    };
  };
}
```

```typescript
// packages/shared-types/src/index.ts — friendly re-exports
export type { components } from './generated';

// Convenience aliases
export type Job = import('./generated').components['schemas']['Job'];
export type JobStatus = import('./generated').components['schemas']['Job']['status'];
export type JobPriority = import('./generated').components['schemas']['Job']['priority'];
export type JobMetrics = import('./generated').components['schemas']['JobMetrics'];
```

### Step 3: Dashboard Uses Shared Types

```typescript
// apps/dashboard/src/types/api.ts — DELETED. No more manual types.

// Instead, import from the shared package:
import type { Job, JobStatus, JobPriority, JobMetrics } from '@job-engine/shared-types';
```

Now when the backend adds `URGENT`, the type generation runs, the shared package updates, and the dashboard gets a TypeScript error: "Type 'URGENT' is not handled in switch statement." The drift is caught at compile time.

### Step 4: Turborepo Configuration

```json
// turbo.json
{
  "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "generate": {
      "inputs": ["../backend/src/main/resources/openapi.yaml"],
      "outputs": ["src/generated.ts"]
    },
    "build": {
      "dependsOn": ["^build", "generate"],
      "outputs": ["dist/**"]
    },
    "test": {
      "dependsOn": ["^build"]
    },
    "lint": {},
    "dev": {
      "cache": false,
      "persistent": true
    }
  }
}
```

```yaml
# pnpm-workspace.yaml
packages:
  - 'apps/*'
  - 'packages/*'
```

```json
// package.json (root)
{
  "name": "job-engine-platform",
  "private": true,
  "scripts": {
    "build": "turbo run build",
    "test": "turbo run test",
    "lint": "turbo run lint",
    "dev": "turbo run dev",
    "generate:types": "turbo run generate --filter=@job-engine/shared-types"
  },
  "devDependencies": {
    "turbo": "^2.0.0"
  }
}
```

### Step 5: The Build Pipeline

```bash
# First run: builds everything
$ pnpm turbo run build
  @job-engine/shared-types:generate  → 0.8s
  @job-engine/shared-types:build     → 1.2s
  @job-engine/dashboard:build        → 8.4s
  Total: 10.4s

# Second run: nothing changed → cache hit
$ pnpm turbo run build
  @job-engine/shared-types:generate  → CACHE HIT (0ms)
  @job-engine/shared-types:build     → CACHE HIT (0ms)
  @job-engine/dashboard:build        → CACHE HIT (0ms)
  Total: 0.3s

# After backend changes openapi.yaml:
$ pnpm turbo run build
  @job-engine/shared-types:generate  → 0.8s (input changed)
  @job-engine/shared-types:build     → 1.2s (dependency changed)
  @job-engine/dashboard:build        → 8.4s (dependency changed)
  Total: 10.4s
```

Turborepo caches task outputs. If nothing changed, the build is instant. If the OpenAPI spec changes, only the affected packages rebuild.

## The Type Safety Chain

```
Java enum changes
  → OpenAPI spec regenerates
  → `pnpm generate:types` runs
  → TypeScript types update
  → Dashboard gets compile error
  → Developer fixes the frontend
  → Types are in sync
```

No manual copying. No "someone should update the frontend types." The pipeline enforces it.

## When to Use a Monorepo

| Situation | Monorepo? | Why |
|-----------|-----------|-----|
| Frontend + backend share types | Yes | Single source of truth |
| Multiple apps share a design system | Yes | One change, all apps update |
| Independent services, different teams | Maybe not | Coordination overhead |
| Solo project, one app | No | Overkill |

## The Result

| Before | After |
|--------|-------|
| Types defined in 2 repos | Types generated from OpenAPI spec |
| Manual sync, always drifting | Auto-generated, always in sync |
| Backend adds field → frontend doesn't know | Backend adds field → TypeScript error |
| 20-minute full rebuild | 10s incremental, 0.3s cached |
| "Did someone update the types?" | Types update themselves |

NullPointer checks the generated types. "The `tags` field is there. The `URGENT` priority is there. Everything matches the Java models."

Bobby Tables checks the API. "The OpenAPI spec is the contract. If it changes, the frontend knows immediately."

Linus nods. "One more thing. Ship it."
